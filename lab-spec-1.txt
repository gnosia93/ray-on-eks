먼저 스케일링부터 정리하면 — 스케일링은 Karpenter만이 아닙니다. 2단(two-layer)입니다.

  - 노드 스케일링 = Karpenter (Pod가 pending이면 노드를 띄움)
  - 애플리케이션 스케일링 = Ray 자체 (RayCluster autoscaler가 worker 수를, Ray Serve가 replica 수를 늘림)

  즉 "Ray가 replica/worker를 더 요구 → Pod pending → Karpenter가 노드 공급" 이렇게 두 층이 맞물려 돌아갑니다.
  Karpenter는 그중 **아래층(노드)**만 담당합니다.

  ---
  리소스 관리 (Resource Management)

  "노드를 몇 대 띄우냐(스케일링)"가 아니라, 이미 확보된 자원을 어떻게 쪼개고, 나눠주고, 못 넘게 막느냐입니다. 핵심
  질문: "이 GPU/CPU/메모리를 누가, 얼마나, 어떤 우선순위로 쓰는가?"

  Ray on EKS 맥락에서 구체적으로:

  ┌────────────────────┬────────────────────────────────────────────────────────┬────────────────────────────┐
  │        항목        │                          의미                          │         이 랩에서          │
  ├────────────────────┼────────────────────────────────────────────────────────┼────────────────────────────┤
  │ Requests / Limits  │ Pod가 요구/상한하는 CPU·메모리·GPU. 이게 있어야        │ RayJob/Serve 스펙의        │
  │                    │ 스케줄링·과금·격리가 성립                              │ resources.limits           │
  ├────────────────────┼────────────────────────────────────────────────────────┼────────────────────────────┤
  │ Fractional GPU     │ GPU 1장을 여러 모델이 나눠 씀 (num_gpus: 0.5). 작은    │ Lab 5 (Serve)              │
  │                    │ 서빙 모델에 GPU 통째로 주면 낭비                       │                            │
  ├────────────────────┼────────────────────────────────────────────────────────┼────────────────────────────┤
  │ 테넌트 격리 / 쿼터 │ 고객 A가 고객 B 자원을 잠식 못 하게. Namespace         │ 이미 3번 문서에 있음       │
  │                    │ ResourceQuota, taint/toleration, nodeSelector          │                            │
  ├────────────────────┼────────────────────────────────────────────────────────┼────────────────────────────┤
  │ 우선순위 / 큐잉    │ 학습(배치, 미뤄도 됨) vs 서빙(지연 민감)의 우선순위.   │ Kueue 큐, PriorityClass    │
  │                    │ 자원 부족 시 누가 양보하나                             │                            │
  ├────────────────────┼────────────────────────────────────────────────────────┼────────────────────────────┤
  │ 오버커밋 방지 /    │ GPU 노드가 놀지 않게 촘촘히 채우기                     │ Karpenter consolidation +  │
  │ bin-packing        │                                                        │ Ray placement              │
  └────────────────────┴────────────────────────────────────────────────────────┴────────────────────────────┘

  ▎ 스케일링과의 차이: 스케일링은 "파이를 키운다", 리소스 관리는 "파이를 어떻게 자른다". 리소스 관리가 부실하면
  ▎ 노드를 아무리 띄워도 idle GPU 과금이 새고, 한 고객이 클러스터를 독식합니다. (4번 갱스케줄링 문서의 "부분 점유
  ▎ idle 과금" 문제가 바로 리소스 관리 실패 사례입니다.)

  프로덕션 레디니스 (Production-Readiness)

  "돌아간다"와 "고객 트래픽을 24/7 안심하고 받을 수 있다"의 차이입니다. 데모 클러스터를 실제 서비스로 승격시킬 때
  반드시 채워야 하는 항목들:

  ┌────────────┬──────────────────────────────────────────┬───────────────────────────────────────────────────┐
  │    영역    │                   질문                   │                     구체 수단                     │
  ├────────────┼──────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ 가용성 /   │ 노드·Pod가 죽어도 서비스가 유지되나?     │ Ray Serve replica 다중화, health check,           │
  │ 복원력     │ Spot이 회수되면?                         │ readiness/liveness probe, 체크포인트 복구         │
  ├────────────┼──────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ 무중단     │ 새 모델 버전을 트래픽 끊김 없이          │ Ray Serve rolling update, 카나리/블루-그린        │
  │ 배포       │ 교체하나? 롤백은?                        │                                                   │
  ├────────────┼──────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ 관측성     │ 장애·지연을 배포 전에 감지하나?          │ Prometheus/Grafana (GPU 사용률, p99 지연, 큐      │
  │            │                                          │ 대기), 로깅, 알림/SLO                             │
  ├────────────┼──────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ 보안 /     │ 고객 데이터·엔드포인트가 서로 새지 않나? │ Namespace 격리, NetworkPolicy, IRSA(IAM), Secret  │
  │ 격리       │                                          │ 관리, private 서브넷                              │
  ├────────────┼──────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ 트래픽     │ 폭주 시 죽지 않고 버티나?                │ 오토스케일 + 요청 큐잉/백프레셔, rate limit, ALB  │
  │ 관리       │                                          │ health check                                      │
  ├────────────┼──────────────────────────────────────────┼───────────────────────────────────────────────────┤
  │ 운영성     │ 새 고객 온보딩이 반복 가능·자동화되어    │ GitOps(ArgoCD) 표준화, 런북, IaC                  │
  │            │ 있나? 장애 대응 절차는?                  │                                                   │
  └────────────┴──────────────────────────────────────────┴───────────────────────────────────────────────────┘

  ▎ 핵심: 프로덕션 레디니스는 정상 상황이 아니라 "고장 났을 때"를 다룹니다. 노드가 죽고, Spot이 회수되고, 트래픽이
  ▎ 튀고, 새 버전에 버그가 있는 상황에서도 고객이 눈치 못 채게 하는 것.

  ---
  세 개를 한 문장으로

  - 스케일링 = 필요할 때 자원을 늘리고 줄인다 (Karpenter=노드 / Ray=worker·replica)
  - 리소스 관리 = 확보한 자원을 공정·효율적으로 나눈다 (requests·quota·격리·fractional GPU)
  - 프로덕션 레디니스 = 고장·부하·배포 상황에서도 고객이 안심하게 서비스한다 (복원력·무중단·관측성·보안)

  이 셋이 각각 랩에 매핑되면: 스케일링/리소스관리는 Lab 4(갱스케줄링)·Lab 5(Serve), 프로덕션 레디니스는 **Lab
  5·Lab 7(관측성/운영)**에 주로 들어갑니다.
